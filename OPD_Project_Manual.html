OPD Project Manual

This document serves as a comprehensive manual for the Outpatient Department (OPD) management system. It covers the project's architecture, core functionalities, technical stack, security considerations, and deployment procedures.

For end-user instructions, please refer to the OPD User Manual.

1. Project Overview

The OPD project is a web-based application designed to streamline the management of outpatient services within a medical facility. It facilitates appointment scheduling, patient record keeping, user management, and provides different interfaces for various user roles.

1.1. Key Features

  • User Authentication & Authorization (Login, Registration, Password Reset)
  • Role-Based Access Control (Super Admin, Admin, Staff, Doctor, Patient)
  • Appointment Management (Booking, Viewing, Status Updates, Rescheduling)
  • Patient Profile & Medical History Viewing
  • User Account Management (Create, Archive, Restore, Delete)
  • File Uploads (Profile Pictures, Appointment Attachments)
  • Audit Logging for critical actions
  • Basic Reporting and Analytics

2. Technical Stack

  • Backend: Java (Jakarta EE Servlets)
  • Frontend: JavaServer Pages (JSP), JSTL, HTML, CSS, JavaScript
  • Build Tool: Apache Maven
  • Database: MariaDB (local development), PostgreSQL (Render deployment)
  • Web Server: Apache Tomcat (via webapp-runner for Render)
  • External Services: Semaphore (for OTP/SMS)

3. Architecture and Design

The project adheres to the Model-View-Controller (MVC) architectural pattern, a standard for web applications.

3.1. MVC Components

  • Model: Java POJOs (e.g., Patient, Appointment, User) representing data structures, and the database schema.
  • View: JavaServer Pages (JSP) files responsible for rendering the user interface. JSTL is used for dynamic content.
  • Controller: Java Servlets (e.g., StaffAddAppointmentServlet, LoginServlet) handling user requests, processing business logic, interacting with the database, and dispatching to the appropriate view.

3.2. Database Interaction

Database operations are performed directly within the Servlets using JDBC. The DBUtil class manages database connection pooling.

Architectural Improvement Needed: DAO/Service Layer

Currently, database logic is tightly coupled within the Servlets. For improved maintainability, testability, and reduced code duplication, it is highly recommended to introduce a dedicated Data Access Object (DAO) layer and a Service layer.

DAO Layer: Abstracts database operations (CRUD) for specific models (e.g., PatientDAO, AppointmentDAO).

Service Layer: Encapsulates business logic, orchestrating calls to multiple DAOs if necessary.

4. Database Schema (MariaDB / PostgreSQL)

The database schema is designed to manage users, appointments, patient profiles, and related medical data. Key tables include:

  • users: Stores user authentication details (username, password hash, contact, user type).
  • user_profiles: Stores detailed profile information for each user (surname, given name, DOB, address, specialization, etc.). Linked to users via user_id.
  • appointments: Records all appointment details, linking to patients (via booked_by_user_id or relative_id), doctors, and rooms.
  • patient_relatives: Stores information about relatives associated with a patient.
  • appointed_relatives: Manages relationships between patients and appointed relatives (e.g., guardians).
  • rooms: Defines available rooms for appointments.
  • services: Lists available medical services.
  • specializations: Lists medical specializations for doctors.
  • designations: Lists staff designations.
  • audit_logs: Records significant user actions for auditing purposes.
  • notifications: Stores system notifications.
  • daily_reports: Stores paths to generated daily reports.
  • months: A utility table for month names.

Relationships are enforced using foreign key constraints to maintain data integrity.

5. Key Functionalities and User Roles

5.1. User Authentication & Authorization

  • Login: Users authenticate with username and password. Password hashes are stored securely.
  • Registration: New patient accounts can be created.
  • Password Reset: Utilizes OTP via Semaphore for secure password recovery.
  • Role-Based Access: Access to Servlets and JSPs is restricted based on userType (Super Admin, Admin, Staff, Doctor, Patient) stored in the session.

Security Improvement Needed: Centralized Authorization

Authorization checks are currently performed at the beginning of each Servlet. This approach is prone to errors and difficult to maintain. Implementing a jakarta.servlet.Filter for centralized authentication and authorization is strongly recommended.

5.2. Appointment Management

  • Booking: Staff can book appointments for new or existing patients, assigning doctors and rooms.
  • Viewing:
    • Patients: View their own upcoming and past appointments.
    • Doctors: View appointments assigned to them.
    • Staff: View all appointments, filter by status, and manage details.
  • Status Updates: Appointments can be marked as Pending, Assigned, Accepted, Rejected, or Completed.
  • Rescheduling: Functionality to propose new dates for appointments.

5.3. Patient Records

  • Staff/Doctors: Can view detailed patient profiles and their medical/appointment history.
  • Patients: Can view their own profile and medical records.
  • Relatives: Can view medical records of patients they are appointed to.

5.4. User Management (Super Admin / Admin)

  • Create Accounts: Super Admins can create Admin, Staff, and Doctor accounts. Admins can create Staff and Doctor accounts.
  • Archive/Restore: Users can be archived (deactivated) and restored.
  • Permanent Delete: Users can be permanently removed from the system.

5.5. File Uploads

  • Profile Pictures: Users can upload profile pictures.
  • Appointment Attachments: Patients can upload relevant documents for appointments.

Critical for Cloud Deployment: Persistent Storage

For cloud deployments (like Render), the application's local filesystem is ephemeral. All uploaded files will be lost on redeployment or restart. A persistent storage solution (e.g., Render Disk, AWS S3, Google Cloud Storage) is mandatory.

6. Deployment to Render

Deploying the project to Render involves adapting the application for a PostgreSQL database and persistent storage.

6.1. Pre-Deployment Steps

  1. Add PostgreSQL JDBC Driver: Include org.postgresql:postgresql dependency in pom.xml.
  2. Update DBUtil.java: Modify DBUtil to parse Render's DATABASE_URL environment variable for PostgreSQL connections, while retaining local MariaDB fallback.
  3. Handle File Uploads: Change file save paths in Servlets to use a Render Disk's mount path (e.g., /var/data/uploads).
  4. Externalize Secrets: Ensure sensitive data like SEMAPHORE_API_KEY and database passwords are read from environment variables (e.g., via .env locally, directly on Render).

6.2. Render Configuration

  1. Git Repository: Push your project to GitHub or GitLab.
  2. Render Web Service: Create a new Web Service on Render.
    • Runtime: Docker
    • Build Command: mvn clean install
    • Start Command: java -jar target/dependency/webapp-runner.jar --port $PORT target/*.war
  3. Add webapp-runner: Include the maven-dependency-plugin in pom.xml to copy webapp-runner.jar.
  4. Render PostgreSQL Database: Create a new PostgreSQL service on Render and copy its Internal Connection URL.
  5. Set Environment Variables: Configure DATABASE_URL (from Render PostgreSQL) and SEMAPHORE_API_KEY in the Web Service's environment settings.
  6. Attach Render Disk: Create and attach a Disk (e.g., named opd-uploads, mounted at /var/data/uploads) to your Web Service for persistent file storage.
  7. Initial Database Setup: After deployment, connect to the Render PostgreSQL database and execute the PostgreSQL-compatible schema script to create tables and populate initial data.

7. Future Enhancements

  • Implement DAO and Service Layers: Decouple database logic from Servlets.
  • Centralized Security Filter: Implement a jakarta.servlet.Filter for robust and maintainable authentication/authorization.
  • Improved Logging: Integrate a logging framework (e.g., SLF4J with Logback) for better error tracking and debugging.
  • Modern Frontend Framework: Consider using a JavaScript framework (React, Angular, Vue) for a more dynamic and responsive user interface, interacting with the backend via REST APIs.
  • Asynchronous Operations: For long-running tasks (e.g., report generation), consider asynchronous processing.
  • Comprehensive Unit/Integration Tests: Develop tests for backend logic and database interactions.
